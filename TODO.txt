AMAZON-M2 NEXT PRODUCT RECOMMENDATION - PROJECT TODO
========================================================

PROJECT OVERVIEW
----------------
Final project for grad course. Task: next product recommendation on the
Amazon-M2 session-based shopping dataset. Rubric: Novelty 20%, Implementation
30%, Experiments 30%, Presentation 20%. Need 2-3 baselines + one novel method.
Novelty matters more than outperformance.

Dataset: ~3.6M training sessions, ~1.4M products, 6 locales (UK, DE, JP large;
ES, FR, IT small). Sessions average ~4 items. Target metrics: MRR@100, Recall@100.
Reference project (Amazon-M2-Session-Recommendation/) exists locally but gitignored.

Framework: RecBole (PyTorch-based recommendation library). Baselines use
built-in RecBole models; novel method will subclass SequentialRecommender.


PAPER-DERIVED CONTEXT (Amazon-M2.pdf, NeurIPS 2023 Datasets & Benchmarks)
------------------------------------------------------------------------
Section 4.1 / Table 3 defines Task 1 (our task). Key facts to cite and plan
against:

- Task 1 baselines reported in paper: Popularity, GRU4Rec++, NARM, STAMP,
  SRGNN, CORE, MGS. Our plan (GRU4Rec, NARM, Popularity + novel) maps
  directly onto a subset.
- Paper Task 1 evaluation is UK/DE/JP only (the three large locales).
  Paper's "Overall" column is the union of those three. Our current
  preprocess/evaluation uses all 6 locales -> our "Overall" is not
  directly comparable to the paper's "Overall" without a locale filter.
- Paper reports MRR@100 and Recall@100 per-locale (UK, DE, JP) plus
  Overall. NDCG@100 also reported in Appendix C. Our evaluate.py
  currently emits only a single global MRR/Recall pair.
- Table 3 headline: Popularity beats every deep model on MRR@100
  (0.2426 vs ~0.15-0.22). Deep models only dominate on Recall@100,
  with CORE leading at 0.5591. GRU4Rec++ in paper = 0.1757 / 0.3808;
  our plain GRU4Rec = 0.1583 / 0.3759 (pre-seed split) - in the right
  ballpark, slightly below "++" which has data augmentation we don't.
- Table 5 ablation: naive Multilingual Sentence-BERT embeddings of
  product titles added to SRGNN / GRU4Rec++ often HURT performance.
  Paper text: "how to effectively utilize textual features remains
  an open question." This is the most citable motivation for any
  attribute-aware novel method.
- Figure 2 dataset-analysis plots: (a) long-tail product frequency,
  (b) cross-locale product overlap heatmap, (c) session length
  distribution, (d) repeat-pattern ratio per locale (~35% across
  all locales), (e) distribution of repeated products per session,
  (f) 10th-highest SKNN score distribution.
- Section 3 SKNN analysis: most sessions can find ~10 similar
  sessions with high cosine-overlap SKNN score. Strong latent
  collaborative-filtering signal a hybrid novel method could use.
- Task 2 (Section 4.2): pretraining on large locales (UK/DE/JP),
  finetuning on small locales (ES/FR/IT). Cross-locale transfer is
  called out as an explicit open problem. Possible novelty angle
  but heavier scope.

NOVEL METHOD:
[ ] Design the novel contribution. Ideas ranked by how well the
    paper motivates them:
    - Text-attribute integration that goes beyond naive BERT
      averaging. Table 5 of the paper shows naive Multilingual
      Sentence-BERT embeddings of titles ADDED to SRGNN /
      GRU4Rec++ often degrades performance. A principled fusion
      (e.g. gated, attention-over-attributes, contrastive
      alignment) has an open-problem citation straight from the
      authors.
    - Popularity-aware deep model. Paper Table 3 shows Popularity
      dominates every deep model on MRR. A hybrid that blends a
      popularity prior with a sequence model at inference (e.g.
      logit interpolation, gated routing) has strong empirical
      motivation.
    - Repeat-consumption modeling. Paper Section 3 + Figure 2d
      show ~35% of sessions contain repeats. Explicitly modeling
      whether the next item is a repeat vs. a new item (two-
      headed decoder, copy mechanism) is a clean, testable design.
    - SKNN-augmented sequence model. Paper Figure 2f shows most
      sessions have high-similarity neighbors. Adding an SKNN
      retrieval branch that conditions the deep model is
      data-driven.
    - Cross-locale transfer (Task 2 framing). Heaviest scope but
      the paper calls it out explicitly; a lightweight adapter or
      shared-embedding scheme could work.

KNOWN ISSUES / GOTCHAS
----------------------
- PyTorch 2.6 changed torch.load default to weights_only=True, which breaks
  RecBole's checkpoint loading. Monkey-patch torch.load in any script that
  triggers a torch.load call (train.py and evaluate.py both need it).
- RecBole 1.2.0 needs numpy<1.24 -- evaluator/metrics.py:90 uses
  np.float (removed in NumPy 1.24) and data_utils uses np.float_
  (removed in NumPy 2.0). Training will run under numpy<2.0 because
  the evaluator code path never fires; the first actual .evaluate()
  call is what trips it. Pin <1.24 to cover both.
- RecBole has undeclared deps: ray, kmeans-pytorch, setuptools (pkg_resources)
- RecBole's SequentialDataset.data_augmentation() wants a timestamp field; we
  bypass this by using pre-split files via benchmark_filename
- ~/.local site-packages can override conda env -- use PYTHONNOUSERSITE=1
  at runtime AND on every pip install line (otherwise pip's
  "already satisfied" check sees ~/.local stale copies and skips
  installing into the env)
- create_env.sh MUST do `module purge && module load Anaconda3/2025.06-0`
  before the conda shell hook. Running it without the module activates a
  different conda with a different envs_dirs, so the sbatch's
  `conda activate amazon-m2-venv` ends up pointing at a different env
  with the same name.
- The "studies" shared filesystem reports False from os.access(path,
  os.W_OK) even when writes actually succeed. Pip interprets that as "not
  writable" and defaults to --user (i.e. ~/.local). Every pip install in
  create_env.sh uses `--no-user` to force the install into the env anyway.
- Mac dev only; actual training must run on the GPU cluster
- 3.6M sessions + 1.4M items means OOM on laptop; always test with --nrows
- run_training.sbatch requires a model argument; submit via
  submit_training.sh or with `sbatch --job-name=<name> run_training.sbatch
  <ModelName>`. Submitting with no args exits 2.
- Paper Task 1 evaluation scope is UK/DE/JP only, so paper "Overall"
  numbers are NOT the same universe as our all-6-locales "Overall".
  Always label evaluation scope when comparing.
- Amazon-M2 Task 1 format: sessions_train.csv packages each session
  as (prev_items, next_item) where next_item is the held-out last
  click and prev_items is everything before it. They are disjoint on
  every row by construction -- verified empirically (0 of 360,625
  test targets appear in their own history). This invalidates any
  naive "session-aware Pop" baseline and blocks reproduction of
  paper Table 3's Popularity MRR on our 80/10/10 split. Paper's
  protocol must differ (likely the official KDD Cup task1 test set
  or a leave-one-out split). Do NOT spend effort chasing the 0.24
  MRR number unless willing to change the evaluation protocol.
- RecBole's Pop is a GeneralRecommender and is INCOMPATIBLE with
  our sequential config (benchmark_filename, alias_of_item_id,
  session_id as USER_ID_FIELD). Training "succeeds" but produces
  garbage popularity counts (~0.0002 MRR). Use scripts/pop_baseline.py
  instead -- hand-rolled, independent of RecBole.
- Interactive-shell `python` may NOT be the venv python even when the
  venv prompt prefix is showing. On the cluster, if the Anaconda3
  module has been loaded in the same shell, `which python` can resolve
  to /opt/apps/easybuild/.../Anaconda3/2025.06-0/.../python3.13 while
  the venv (amazon-m2-venv) is python 3.10. Parquet files written by
  the venv's pyarrow may then fail to read under the system pyarrow
  with "OSError: Repetition level histogram size mismatch" (pyarrow
  version drift between writer and reader, not actual corruption).
  Fix: force the venv binary explicitly
  (/media/studies/ehr_study/analysis/mferguson/venvs/amazon-m2-venv/bin/python)
  or re-activate cleanly: `module purge && module load
  Anaconda3/2025.06-0 && conda activate amazon-m2-venv` and re-check
  `which python`.
