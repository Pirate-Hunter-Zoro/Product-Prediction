AMAZON-M2 NEXT PRODUCT RECOMMENDATION - PROJECT TODO
========================================================

PROJECT OVERVIEW
----------------
Final project for grad course. Task: next product recommendation on the
Amazon-M2 session-based shopping dataset. Rubric: Novelty 20%, Implementation
30%, Experiments 30%, Presentation 20%. Need 2-3 baselines + one novel method.
Novelty matters more than outperformance.

Dataset: ~3.6M training sessions, ~1.4M products, 6 locales (UK, DE, JP large;
ES, FR, IT small). Sessions average ~4 items. Target metrics: MRR@100, Recall@100.

Framework: RecBole (PyTorch-based recommendation library). Baselines use
built-in RecBole models; novel method will subclass SequentialRecommender.


PAPER-DERIVED CONTEXT (Amazon-M2.pdf, NeurIPS 2023 Datasets & Benchmarks)
------------------------------------------------------------------------
Section 4.1 / Table 3 defines Task 1 (our task). Key facts to cite and plan
against:

- Task 1 baselines reported in paper: Popularity, GRU4Rec++, NARM, STAMP,
  SRGNN, CORE, MGS. Our plan (GRU4Rec, NARM, Popularity + novel) maps
  directly onto a subset.
- Paper Task 1 evaluation is UK/DE/JP only (the three large locales).
  Paper's "Overall" column is the union of those three. Our current
  preprocess/evaluation uses all 6 locales -> our "Overall" is not
  directly comparable to the paper's "Overall" without a locale filter.
- Paper reports MRR@100 and Recall@100 per-locale (UK, DE, JP) plus
  Overall. NDCG@100 also reported in Appendix C. Our evaluate.py
  currently emits only a single global MRR/Recall pair.
- Table 3 headline: Popularity beats every deep model on MRR@100
  (0.2426 vs ~0.15-0.22). Deep models only dominate on Recall@100,
  with CORE leading at 0.5591. GRU4Rec++ in paper = 0.1757 / 0.3808;
  our plain GRU4Rec = 0.1583 / 0.3759 (pre-seed split) - in the right
  ballpark, slightly below "++" which has data augmentation we don't.
- Table 5 ablation: naive Multilingual Sentence-BERT embeddings of
  product titles added to SRGNN / GRU4Rec++ often HURT performance.
  Paper text: "how to effectively utilize textual features remains
  an open question." This is the most citable motivation for any
  attribute-aware novel method.
- Figure 2 dataset-analysis plots: (a) long-tail product frequency,
  (b) cross-locale product overlap heatmap, (c) session length
  distribution, (d) repeat-pattern ratio per locale (~35% across
  all locales), (e) distribution of repeated products per session,
  (f) 10th-highest SKNN score distribution.
- Section 3 SKNN analysis: most sessions can find ~10 similar
  sessions with high cosine-overlap SKNN score. Strong latent
  collaborative-filtering signal a hybrid novel method could use.
- Task 2 (Section 4.2): pretraining on large locales (UK/DE/JP),
  finetuning on small locales (ES/FR/IT). Cross-locale transfer is
  called out as an explicit open problem. Possible novelty angle
  but heavier scope.

[x] Chose novel-method direction (2026-04-22): cross-attention over
    per-item attributes. Query = session's ID-based sequence
    representation (from a GRU or transformer encoder over the
    item_id_list). Keys/values = a small bag of per-item attribute
    tokens (title embedding + brand/price/color embeddings). Four
    attribute "slots" per item. Motivation: paper Table 5 shows
    naive multilingual Sentence-BERT title embeddings added to
    SRGNN/GRU4Rec++ often HURT performance; paper text explicitly
    calls effective text-feature fusion an open problem. Principled
    fusion via cross-attention has citable open-problem motivation
    straight from the authors. Alternatives rejected:
    - Popularity-aware hybrid: our Pop MRR is 0.0001 -- no
      signal to blend, dead on our split.
    - Repeat-consumption: our Task 1 format holds next_item
      disjoint from prev_items on every row; a copy mechanism
      would copy zero targets.
    - SKNN-augmented sequence model: heavier scope (retrieval
      index over 3.6M sessions), no catalog-parity gain.
    - Cross-locale Task 2: different task entirely, scope creep
      masquerading as novelty.
[x] scripts/build_item_attributes.py (2026-04-22):
    - Reads data/products_train.csv (catalog dump, one row per
      (id, locale)), keeps five columns (id, title, brand, price,
      color), renames id -> item_id, fillna "" on the three text
      columns. locale/size/model/material/author/desc dropped for
      this first pass; revisit if cross-attention needs more
      attribute slots.
    - Builds item vocab from the union of item_id:token and
      exploded item_id_list:token_seq columns across
      amazon_m2.{train,valid,test}.inter; filters to string members
      only via {v for v in vocab if isinstance(v, str)} to drop
      NaN and empty-string sentinels that .explode() can leave
      behind. Note: set.discard(float("nan")) does NOT work because
      float("nan") != float("nan") -- use the isinstance filter.
    - Filters products to rows whose item_id is in vocab, then
      drop_duplicates(subset="item_id", keep="first") to collapse
      per-locale duplicate rows down to one row per SKU.
      keep="first" picks an arbitrary-but-deterministic locale;
      multilingual handling (per-locale titles) is a future
      decision if cross-attention encoding needs it.
    - Writes data/amazon_m2/item_attributes.parquet:
      item_id:object, title:object, brand:object, price:float64,
      color:object. 1,393,956 rows. Zero nulls in any column.
      Empty-string counts post-fillna: title 8, brand 18,923,
      color 395,510 (legitimate catalog sparsity, not a bug).
    - CLI: --products-csv, --inter-dir, --output, --nrows.
      Defaults align with the rest of the pipeline.
    - KEY FINDING: catalog_gap = len(vocab) -
      filtered_products["item_id"].nunique() = 0. Every item
      referenced in any session has a catalog row. Amazon-M2
      authors curated the KDD Cup 2023 release to be internally
      consistent -- they did not ship sessions containing items
      whose catalog metadata they withheld. CONSEQUENCE: the novel
      cross-attention model needs NO <UNK> attribute fallback, no
      out-of-vocab branch, no imputation policy. Every item the
      sequence encoder will ever see has title/brand/price/color
      available.
    - Diagnostic counterpart (Direction A): len(products) -
      len(filtered_products) = 157,101 catalog rows with no
      session appearance. Expected long-tail behavior; dropped by
      the .isin(vocab) filter so the parquet only contains items
      the model will actually rank.
    - Bug gotchas found during construction, recorded for future
      editors:
      * pandas 2.x: df[col].fillna(v, inplace=True) operates on a
        copy and triggers ChainedAssignmentError / FutureWarning.
        Use df[[cols]] = df[[cols]].fillna(v) instead -- no
        inplace kwarg.
      * df[[cols]] = df[[cols]].fillna(v, inplace=True) is worse
        than useless: fillna(inplace=True) returns None, so the
        RHS is None and the three columns get assigned to NaN
        wholesale. Silent data loss with only a SettingWithCopyWarning
        to warn you.
      * f-strings with nested double-quoted subscripts
        (f"{d["k"]}") are a SyntaxError on Python 3.10/3.11
        (PEP 701 only lifted this restriction in 3.12). Our env
        is 3.10; use single quotes inside.
[x] scripts/encode_titles.py + run_encode_titles.sbatch (2026-04-22):
    - Two functions. load_titles(parquet_path) reads

      via torch.save. Callsite passes args.device directly; no
      "cuda if available" silent-fallback so a missing GPU crashes

        Vocab size:            1,356,886
        Train prices shape:    (1356886,), dtype float64
        Boundaries shape:      (31,); min 3.78, median 24.49, max 40000000.07
        Bin indices shape:     (1393956,); min 0, max 31
        Bin occupancy:         min 17,977, max 69,593

    - Bin balance: observed 2.7x spread (17,977 to 69,593) against
      expected 43,561 per bin. NOT a bug. Retail prices are not a
      smooth continuum -- they pile up on psychological price
      points ($9.99, $14.99, $19.99, etc.). When a quantile cut
      lands on or near one of those point masses, the mass does
      not split; all tied items land in one bin, which forces
      adjacent bins to shrink. Min bin count > 0 confirms all
      31 cut points are distinct (no collapsed bins). A dedicated
      "break ties with noise" pre-step would smooth this out but
      would destroy the interpretability of the bin embedding by
      putting same-price items into different bins. Rejected.
    - DATA QUALITY FINDING: the 31/32 quantile boundary is
      $40,000,000.07. At 1.35M train items this means ~42,000 items
      are priced above $40M in products_train.csv. Amazon does not
      sell forty-million-dollar products -- almost certainly
      placeholder / sentinel / junk values in the raw catalog.
      Mitigation: NONE required for the novel model. Because we
      bucketize, every absurdly-high price collapses into the top
      bin (bin_idx=31), whose 384-dim embedding is learned end-to-
      end by the recommendation loss. Gradients do not see the
      raw dollar figure and cannot be destabilized by an outlier
      magnitude; the top bin simply learns a "weird-price" latent
      signal alongside the other 31 bins. Bucketize-then-embed is
      specifically robust to this class of data contamination,
      whereas a direct numeric embedding (e.g. log1p + linear
      projection) would have been pulled by the $40M tail. Cite
      this in the report when discussing the price encoder choice.
[x] scripts/attribute_loader.py (2026-04-23):
    - Two pure functions, no CLI, no main. This file is imported by
      the model; it is not an entry point.
    - Module top: monkey-patches torch.load with weights_only=False
      via kwargs.setdefault so a caller can still override. Same
      category of patch as train.py and evaluate.py, needed here
      because the attribute pickles contain Python list[str] which
      PyTorch 2.6's safe-load allowlist rejects.
    - load_text_embedding(pickle_path, dataset, item_id_field="item_id")
      -> FloatTensor[num_items, embedding_dim]. Reads a
      {title,brand,color}_embeddings.pt pickle (dict with keys
      "item_ids": list[str] and "embeddings": FloatTensor[N, 384]),
      allocates a zero output tensor sized by dataset.num(item_id_field)
      so row 0 is the PAD slot, batched-translates every external
      token to its RecBole internal ID via dataset.token2id (single
      vectorized call, NOT a Python loop over 1.39M items), and does
      one index-assignment output[internal_ids] = embeddings. Row 0
      stays zero. Embedding dim is read from embeddings.shape[1] --
      not hardcoded to 384 -- so a future encoder swap does not
      require editing the loader.
    - load_price_bins(pickle_path, dataset, item_id_field="item_id",
      n_bins=32) -> LongTensor[num_items]. Reads price_bins.pt (dict
      with "item_ids": list[str] and "bin_idx": LongTensor[N]),
      asserts bin_idx.dtype == torch.long (loud localized failure if
      bucketize_price.py's dtype is ever "fixed" to int32; nn.Embedding
      would otherwise crash far downstream with a confusing error),
      allocates torch.full((num_items,), n_bins, dtype=torch.long) so
      every row starts at the reserved sentinel, batched-translates
      tokens via token2id, and scatters bin indices into their internal
      ID rows. Row 0 (PAD) stays at n_bins; any hypothetical vocab/pickle
      desync also degrades to n_bins rather than silently landing items
      in bin 0. Caller's nn.Embedding must therefore be sized n_bins + 1
      to host the reserved PAD embedding.

      item_ids list is in the same row order as the embeddings /
      bin_idx tensor it ships with, because encode_text_attribute.py
      and bucketize_price.py both preserve parquet row order. If this
      invariant ever breaks on the writer side, the index-assignment
      on the reader side will silently produce mis-aligned embeddings
      with no crash -- the invariant is load-bearing.
    - Catalog gap = 0 for this pipeline (see build_item_attributes.py
      completed entry), so the sentinel path in load_price_bins is

    Option B (session-conditioned attention over the session's own
    items, NOT full vocab). The literal TODO reading -- query =
    session state, K/V = full attribute bag -- would have allocated
    ~734 GB of KV at batch=512, hidden=64; impossible. Option B

[x] predict(interaction) method added to scripts/novel_model.py
    (2026-04-27). Inserted between calculate_loss and full_sort_predict.
    Body:
      1. item_seq = interaction[self.ITEM_SEQ] -> (batch, max_seq_len)
      2. item_seq_lengths = interaction[self.ITEM_SEQ_LEN] -> (batch,)
      3. item_candidates = interaction[self.ITEM_ID] -> (batch,)
         -- NOT self.POS_ITEM_ID. predict scores arbitrary candidates;
         POS_ITEM_ID would only work during training where the target
         is known.
      4. sequence_embeddings = self.forward(item_seq, item_seq_lengths)
         -> (batch, hidden_size)
      5. candidate_embeddings = self.item_embedding(item_candidates)
         -> (batch, hidden_size)
      6. return torch.sum(sequence_embeddings * candidate_embeddings,
                          dim=1) -> (batch,)
    Same row-wise dot-product scoring rule used by calculate_loss
    (sum-reduced) and full_sort_predict (matmul against full vocab).
    Consistency across the three RecBole entry points is required:
    if predict scored differently, get_flops would report wrong
    parameter counts AND any sampled-candidate eval mode would
    disagree with full-sort.
    No torch.no_grad() / model.eval() inside the body -- the trainer
    wraps eval calls in both already, and hard-coding either inside
    predict would break the get_flops caller (which builds an
    autograd graph) and any future gradient-needing caller.

KNOWN ISSUES / GOTCHAS
----------------------
- PyTorch 2.6 changed torch.load default to weights_only=True, which breaks
  RecBole's checkpoint loading. Monkey-patch torch.load in any script that
  triggers a torch.load call (train.py and evaluate.py both need it).
- RecBole 1.2.0 needs numpy<1.24 -- evaluator/metrics.py:90 uses
  np.float (removed in NumPy 1.24) and data_utils uses np.float_
  (removed in NumPy 2.0). Training will run under numpy<2.0 because
  the evaluator code path never fires; the first actual .evaluate()
  call is what trips it. Pin <1.24 to cover both.
- RecBole has undeclared deps: ray, kmeans-pytorch, setuptools (pkg_resources)
- RecBole's SequentialDataset.data_augmentation() wants a timestamp field; we
  bypass this by using pre-split files via benchmark_filename
- ~/.local site-packages can override conda env -- use PYTHONNOUSERSITE=1
  at runtime AND on every pip install line (otherwise pip's
  "already satisfied" check sees ~/.local stale copies and skips
  installing into the env)
- create_env.sh MUST do `module purge && module load Anaconda3/2025.06-0`
  before the conda shell hook. Running it without the module activates a
  different conda with a different envs_dirs, so the sbatch's
  `conda activate amazon-m2-venv` ends up pointing at a different env
  with the same name.
- The "studies" shared filesystem reports False from os.access(path,
  os.W_OK) even when writes actually succeed. Pip interprets that as "not
  writable" and defaults to --user (i.e. ~/.local). Every pip install in
  create_env.sh uses `--no-user` to force the install into the env anyway.

- Amazon-M2 Task 1 format: sessions_train.csv packages each session
  as (prev_items, next_item) where next_item is the held-out last
  click and prev_items is everything before it. They are disjoint on
  every row by construction -- verified empirically (0 of 360,625
  test targets appear in their own history). This invalidates any
  naive "session-aware Pop" baseline and blocks reproduction of
  paper Table 3's Popularity MRR on our 80/10/10 split. Paper's
  protocol must differ (likely the official KDD Cup task1 test set
  or a leave-one-out split). Do NOT spend effort chasing the 0.24
  MRR number unless willing to change the evaluation protocol.
- RecBole's Pop is a GeneralRecommender and is INCOMPATIBLE with

  session_id as USER_ID_FIELD). Training "succeeds" but produces

- Interactive-shell `python` may NOT be the venv python even when the
  venv prompt prefix is showing. On the cluster, if the Anaconda3
  module has been loaded in the same shell, `which python` can resolve
  to /opt/apps/easybuild/.../Anaconda3/2025.06-0/.../python3.13 while
  the venv (amazon-m2-venv) is python 3.10. Parquet files written by
  the venv's pyarrow may then fail to read under the system pyarrow
  with "OSError: Repetition level histogram size mismatch" (pyarrow
  version drift between writer and reader, not actual corruption).
  Fix: force the venv binary explicitly
  (/media/studies/ehr_study/analysis/mferguson/venvs/amazon-m2-venv/bin/python)
  or re-activate cleanly: `module purge && module load
  Anaconda3/2025.06-0 && conda activate amazon-m2-venv` and re-check
  `which python`.
- argparse type=bool is a landmine and does not do what the name
  suggests. `parser.add_argument('--flag', type=bool, default=False)`
  creates a flag that CONSUMES A VALUE -- `--flag` alone crashes
  argparse with "expected one argument". And bool("False") is True
  because any non-empty string is truthy in Python, so `--flag False`
  turns the flag ON. Always use `action='store_true'` (or
  `action='store_false'`) for boolean CLI flags. No type=, no default=.
- products_train.csv contains extreme long-tail price outliers.
  The 31/32 quantile over train prices is $40,000,000.07, implying
  ~42,000 items priced above $40M. Clearly placeholder / sentinel /
  junk values, not real Amazon listings. NOT fixed upstream. Safe
  for the novel model because every absurd price collapses into
  bin_idx=31 under 32-quantile bucketing, and that bin's learnable
  384-dim embedding is updated by the recommendation loss --
  gradients never see raw dollar figures. A direct numeric
  encoder (e.g. log1p + linear) would have been pulled by the
  $40M tail; bucketize-then-embed is specifically robust here.
  Mention in the report when discussing the price encoder choice.
